# T-0001 — Migrate from NextAuth to BetterAuth

Replace NextAuth v5 (beta) with BetterAuth across the chatbot. The current auth surface is small — email/password credentials + auto-created guest sessions — so migration is bounded.

## Context

NextAuth v5 is still in beta and the project only uses credential-based providers (no OAuth). BetterAuth is a more actively maintained, TypeScript-first auth library with built-in support for email/password, session management, and database adapters — a better fit for this use case.

## Current State

- **NextAuth v5 beta 25** — `next-auth@5.0.0-beta.25`
- **Two credential providers**: email/password (bcrypt-ts) and guest (auto-created anonymous users)
- **JWT sessions** with custom claims (`id`, `type: "guest" | "regular"`)
- **Middleware** (`src/proxy.ts`): validates tokens via `getToken()`, redirects unauthenticated users to guest signup
- **DB schema**: `user` table with `id`, `email`, `password`, `isAnonymous`, timestamps (Drizzle + Postgres)
- **Server actions**: `login()` and `register()` in `src/app/(auth)/actions.ts` using Zod validation

## Scope

1. **Install BetterAuth** — `better-auth` + Drizzle adapter
2. **Configure BetterAuth server** — replace `auth.ts` with BetterAuth server instance (email/password plugin, session config)
3. **Configure BetterAuth client** — create client-side auth helper for React
4. **Migrate DB schema** — generate BetterAuth tables or adapt existing `user` table; write a Drizzle migration
5. **Migrate guest flow** — replicate guest auto-creation (anonymous user with `isAnonymous` flag) using BetterAuth's API
6. **Migrate middleware** — replace `getToken()` with BetterAuth session validation in `src/proxy.ts`
7. **Update all auth consumers** — swap `auth()` calls for BetterAuth session retrieval across all API routes and server actions
8. **Update login/register pages** — adapt forms and server actions to use BetterAuth sign-in/sign-up
9. **Remove NextAuth** — delete `next-auth` and `@auth/*` packages, remove old config files
10. **Migration for existing users** — ensure existing password hashes (bcrypt-ts) are compatible or provide a rehash path

**Key notes from docs:**

- Route handler goes in `app/api/auth/[...all]/route.ts` — use `toNextJsHandler(auth)` to export GET/POST
- Anonymous plugin adds `isAnonymous` field to user table automatically — maps directly to our existing field
- Anonymous users can link accounts later via `authClient.anonymous.linkTo()` — useful if we want guest→registered upgrade
- Drizzle adapter: `@better-auth/drizzle-adapter` — pass `drizzle(db)` instance + schema. Can auto-generate schema with `npx auth generate --output ./src/lib/db/auth-schema.ts`
- Custom password hashing supported via `password.hash` and `password.verify` options — we can plug in bcrypt-ts for backwards compat
- Migration guide recommends running both systems in parallel briefly, but our surface is small enough to swap in one pass

**Session strategy: stateless cookie cache (no DB sessions)**

Use cookie cache with `refreshCache: true` — same zero-DB-hit model as current NextAuth JWT setup. Config:

```ts
session: {
    cookieCache: {
        enabled: true,
        maxAge: 7 * 24 * 60 * 60, // 7 days
        strategy: "jwe",           // encrypted, tamper-proof
        refreshCache: true,        // auto-refresh at 80% of maxAge, no DB hit
    },
},
```

This eliminates the need for a `session` table and keeps middleware edge-compatible (local cookie verification, no TCP to Postgres). Tradeoff: no per-session revocation — same as current JWT setup. Can add Redis secondary storage later if needed.

**`UserType` removal:** Replace `session.user.type` ("guest" | "regular") with a direct `user.isAnonymous` check. Only two consumers: `entitlements.ts` and `chat/route.ts`. Use `customSession` plugin to expose `isAnonymous` on the session object if needed.
