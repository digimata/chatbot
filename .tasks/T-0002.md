# T-0002 — Integrate Chat SDK for multi-platform bot support

Add [Chat SDK](https://chat-sdk.dev) (`chat` package) to expose the chatbot's AI capabilities on external messaging platforms (Slack, Discord, etc.) alongside the existing web UI.

## Context

The chatbot currently only has a web interface (Next.js app). Chat SDK provides a unified abstraction for receiving and sending messages across Slack, Teams, Discord, Telegram, Google Chat, GitHub, Linear, and WhatsApp — with a single set of event handlers. This would let us deploy the same AI chat logic to external platforms without building per-platform integrations.

## What Chat SDK provides

- **`Chat` class** — main entry point. Configures adapters, routes incoming webhooks to handlers, manages thread state.
- **Platform adapters** — `@chat-adapter/slack`, `@chat-adapter/discord`, `@chat-adapter/teams`, `@chat-adapter/telegram`, `@chat-adapter/google-chat`, `@chat-adapter/github`, `@chat-adapter/linear`, `@chat-adapter/whatsapp`. Each handles webhook verification, message parsing, and API calls.
- **State adapters** — `@chat-adapter/state-redis`, `@chat-adapter/state-postgres`. Persists thread subscriptions, distributed locking, and per-thread state across restarts.
- **Event model** — `onNewMention`, `onSubscribedMessage`, `onReaction`, `onAction` (button clicks), `onModalSubmit`, `onCommand` (slash commands).
- **Thread abstraction** — `thread.post()`, `thread.subscribe()`, `thread.messages`, `thread.state`, `thread.startTyping()`. Normalized across platforms.
- **Rich content** — JSX-like cards, buttons, select menus, modals, tables, file uploads. Automatically converted to each platform's native format (Block Kit, Adaptive Cards, embeds, etc.).
- **Streaming** — native streaming on Slack, post+edit fallback on other platforms. Configurable via `streamingUpdateIntervalMs`.
- **Thread locking** — distributed locks prevent concurrent handler execution per thread. Configurable conflict resolution (`drop` or `force` for interrupt/steerability).

## Integration approach

The existing AI logic lives in `src/lib/ai/` (providers, prompts, tools, models). The Chat SDK integration would:

1. Create a `src/lib/bot.ts` — `Chat` instance with adapter(s) and state
2. Create `src/app/api/webhooks/[platform]/route.ts` — dynamic webhook handler
3. Wire `onNewMention` / `onSubscribedMessage` to call the same `streamText()` pipeline used by `/api/chat`
4. Decide which tools to expose (weather, document creation probably not relevant for Slack — may want a reduced tool set)
5. Stream responses back via `thread.post()` with streaming support

### Key decisions

- **Which platforms first?** Slack is the most feature-complete adapter (native streaming, modals, slash commands, reactions). Start there.
- **State adapter** — we already have Redis (optional) and Postgres. Use whichever is configured.
- **Thread state** — could store conversation history in Chat SDK's thread state, or reuse existing `chat`/`message_v2` tables.
- **Auth mapping** — platform users aren't the same as web users. Need to decide if external platform messages create/link to a chatbot user, or run under a service account.
- **Tool scoping** — artifacts (documents, code, sheets) are web-only. External platforms should get a subset of tools, or tools adapted for that context.

## Scope

1. **Install deps** — `chat`, `@chat-adapter/slack`, `@chat-adapter/state-redis` (or `state-postgres`)
2. **Create bot instance** — `src/lib/bot.ts` with Slack adapter + state
3. **Webhook route** — `src/app/api/webhooks/[platform]/route.ts` using `bot.webhooks`
4. **AI handler** — connect `onNewMention` / `onSubscribedMessage` to existing `streamText()` pipeline with appropriate prompts and tools
5. **Streaming** — use Chat SDK's streaming support for real-time responses
6. **Env vars** — `SLACK_BOT_TOKEN`, `SLACK_SIGNING_SECRET` (+ existing `REDIS_URL`)
7. **Slack app config** — manifest for OAuth scopes, event subscriptions, interactivity

## Done when

- Slack bot responds to @mentions with AI-generated responses
- Subscribed threads get follow-up replies
- Responses stream in real-time (native Slack streaming)
- Thread state persists across restarts
- Existing web UI unaffected
- At least one additional platform adapter works (Discord or Teams) to validate the multi-platform abstraction
